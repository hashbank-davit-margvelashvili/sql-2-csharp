Key insight: In SQL, data and behavior are separate. In C#, they live together inside a class.

From Table to Class

SQL Table (hw.Accounts)

CREATE TABLE hw.Accounts (
    AccountID   BIGINT IDENTITY PK,
    CustomerID  INT,
    AccountNumber VARCHAR(30),
    CCY         VARCHAR(5) DEFAULT 'GEL',
    Balance     DECIMAL DEFAULT 0,
    BlockedAmount DECIMAL DEFAULT 0,
    CreateTime  DATETIME2 DEFAULT SYSDATETIME()
);

C# Class

public class BankAccount
{
    public long AccountId { get; set; }
    public int? CustomerId { get; set; }
    public string? AccountNumber { get; set; }
    public string CCY { get; set; } = "GEL";
    public decimal Balance { get; set; } = 0m;
    public decimal BlockedAmount { get; set; } = 0m;
    public DateTime CreateTime { get; set; }
        = DateTime.Now;
}
A class is CREATE TABLE. An object is a row. Properties are columns. Default values map directly.

Creating Objects — Like INSERT

SQL INSERT

INSERT INTO hw.Accounts
    (CustomerID, AccountNumber, CCY)
VALUES
    (42, 'AC01GEL000000001', 'GEL');
-- Balance, BlockedAmount get defaults

C# Object Initializer

var account = new BankAccount
{
    CustomerId = 42,
    AccountNumber = "AC01GEL000000001",
    CCY = "GEL"
};
// Balance, BlockedAmount get defaults

new is like INSERT — it creates a new instance. Properties you don't set keep their default values, just like columns with DEFAULT.

Data + Behavior Together

In SQL, the function is separate from the table. In C#, the method lives inside the class:

SQL — Separate

-- Table holds data
CREATE TABLE hw.Accounts (
    Balance DECIMAL,
    BlockedAmount DECIMAL
);

-- Function operates on it externally
CREATE FUNCTION
    hp.fnGetAvailableBalance
    (@AccountID BIGINT)
RETURNS DECIMAL AS BEGIN
    SELECT Balance - BlockedAmount
    FROM hw.Accounts
    WHERE AccountID = @AccountID;
END

C# — Together

public class BankAccount
{
    // Data
    public decimal Balance { get; set; }
    public decimal BlockedAmount { get; set; }

    // Behavior — uses its own data
    public decimal GetAvailableBalance()
    {
        return Balance - BlockedAmount;
    }
}

// No query needed — the object HAS the data
var acct = new BankAccount
    { Balance = 1000m, BlockedAmount = 200m };
acct.GetAvailableBalance(); // 800m

Computed Properties — Computed Columns

The hp.CurrencyPairs table has computed columns. In C#, use => properties:

SQL Computed Column

CREATE TABLE hp.CurrencyPairs (
    BaseCurrency  VARCHAR(3),
    QuoteCurrency VARCHAR(3),

    PairCode AS
      (BaseCurrency + '/'
       + QuoteCurrency),

    NormalizedKey AS
      (UPPER(BaseCurrency) + '/'
       + UPPER(QuoteCurrency))
);

C# Computed Property

public class CurrencyPair
{
    public string BaseCurrency { get; set; }
    public string QuoteCurrency { get; set; }

    public string PairCode
        => $"{BaseCurrency}/{QuoteCurrency}";

    public string NormalizedKey
        => PairCode.ToUpper();
}

Computed properties are recalculated every time you access them — always up to date, just like SQL computed columns.

Encapsulation — private set

Control who can modify a property. Read: everyone. Write: only the class.

public class BankAccount
{
    public decimal Balance { get; private set; } = 0m;

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Must be positive.");
        Balance += amount;      // OK — we're inside the class
    }
}

var acct = new BankAccount();
acct.Deposit(500m);                 // OK — uses the method
Console.WriteLine(acct.Balance);    // OK — can read (500)
acct.Balance = 1000m;               // COMPILER ERROR — can't write!
Like a column you can SELECT but can only UPDATE through a stored procedure — controlled modification.

Constructors — The INSERT Rules

A constructor runs when you create a new object. It enforces what data is required:

public class CurrencyCode
{
    public string Code { get; }           // get-only = immutable after construction
    public string NumericCode { get; }
    public int Scale { get; }

    // Constructor — like NOT NULL columns: you MUST provide these values
    public CurrencyCode(string code, string numericCode, int scale)
    {
        if (string.IsNullOrWhiteSpace(code) || code.Length != 3)
            throw new ArgumentException("Code must be 3 characters.");
        Code = code.ToUpper();
        NumericCode = numericCode;
        Scale = scale;
    }
}

var gel = new CurrencyCode("GEL", "981", 2);   // OK
var bad = new CurrencyCode("", "000", 2);      // THROWS — validation!
var nope = new CurrencyCode();                    // COMPILER ERROR — no parameterless constructor
A constructor with required parameters = NOT NULL columns with no DEFAULT. You must provide values at INSERT time.

Constructor Defaults & Chaining

Optional Parameters (Like Column Defaults)

public BankAccount(long accountId, int customerId,
                   string ccy = "GEL", decimal initialBalance = 0m)
{
    AccountId = accountId;  CustomerId = customerId;
    CCY = ccy;  Balance = initialBalance;
}

var a = new BankAccount(1, 42);                    // GEL, 0m
var b = new BankAccount(2, 42, "USD");              // USD, 0m
var c = new BankAccount(3, 42, "EUR", 1000m);     // EUR, 1000m

Constructor Chaining (: this(...))

// Full constructor
public BankAccount(long id, int custId, string ccy, decimal balance)
{
    AccountId = id; CustomerId = custId; CCY = ccy; Balance = balance;
}

// Simplified — chains to the full one
public BankAccount(long id, int custId)
    : this(id, custId, "GEL", 0m) { }   // delegates to full constructor

Quick Check

In SQL, data (tables) and behavior (stored procedures) are separate. In C#, where does behavior live?

A. In a separate static file
B. Inside the class, as methods alongside properties
C. In a stored procedure equivalent called a "handler"
D. In a global functions file
✓ Correct! This is the fundamental OOP shift: data and behavior live together inside a class. The object knows how to operate on its own data.
✗ Not quite. In OOP, methods (behavior) live inside the class, right next to the properties (data). This is the key difference from SQL's approach.

Access Modifiers — GRANT/DENY for C#

C#SQL AnalogyWho Can See It
publicGRANT EXECUTE TO PUBLICAny code, anywhere
privateNo GRANT (owner only)Only inside this class
protectedGRANT to schema familyThis class + subclasses
internalGRANT to database onlyThis project/assembly
public class BankAccount
{
    public  decimal Balance { get; private set; }   // read: public, write: private
    private decimal _overdraftLimit = 0m;             // only this class sees this

    public bool Withdraw(decimal amount)              // public interface
    {
        if (!CanWithdraw(amount)) return false;       // calls private method
        Balance -= amount;
        return true;
    }

    private bool CanWithdraw(decimal amount)           // hidden from outside
        => amount > 0 && amount <= Balance + _overdraftLimit;
}

Encapsulation in Practice

Think of a class like a stored procedure: callers see a simple interface, complexity is hidden inside.

What Outside Code Sees

PUBLIC

Balance (read-only)
AvailableBalance (read-only)
Deposit(amount)
Withdraw(amount)
BlockAmount(amount)

Hidden Inside the Class

PRIVATE

_overdraftLimit
CanWithdraw(amount)
ValidatePositiveAmount()
Balance setter
BlockedAmount setter

Rule of thumb: Start with everything private. Only make things public when another class genuinely needs access.

static — Functions Without a Row

Some SQL functions don't need a table row. C#'s static is the same idea:

SQL Scalar Functions

-- Standalone utilities
SELECT dbo.iban_mod97('1614001821');

SELECT hp.fnCalcFeeAmount(
    1000, 5, 1.0, 0, 100);

-- No table row needed

C# Static Methods

// Call on the CLASS, not an object
int r = IbanValidator.Mod97("1614001821");

decimal fee = FeeCalculator.CalcFeeAmount(
    1000m, 5m, 1.0m, 0m, 100m);

// No object needed
A static method is like a scalar function: dbo.iban_mod97() → IbanValidator.Mod97(). The schema is the class name.

Static Classes — Schema of Functions

public static class FeeCalculator
{
    public static decimal CalcFeeAmount(
        decimal amount, decimal fixedFee, decimal percent,
        decimal min, decimal max)
    {
        decimal fee = fixedFee > 0 ? fixedFee : amount * percent / 100m;
        return Math.Max(min, Math.Min(max, fee));
    }

    public static decimal CalcVat(decimal amount) => amount * 0.18m;
}

When to Use Static

  • Pure calculations (no state)
  • Utility/helper methods
  • Constants and configuration

When to Use Regular Classes

  • Object holds data (properties)
  • Behavior depends on object state
  • Need multiple instances

C# Namespaces

namespace Banking.Core;
namespace Banking.Currency;

// Without using — fully qualified (like hp.fnGenerateAccountNumber)
Banking.Core.BankAccount acct = new Banking.Core.BankAccount();

// With using — like setting a default schema
using Banking.Core;
BankAccount acct = new BankAccount();   // clean!
using Banking.Core; is like ALTER USER SET DEFAULT_SCHEMA = hp; — now you don't need to type the schema prefix.

Quick Check

You want Balance to be readable by any code but only modifiable through Deposit() and Withdraw() methods. Which declaration?

A. public decimal Balance { get; set; }
B. public decimal Balance { get; private set; }
C. private decimal Balance { get; set; }
D. public decimal Balance { private get; set; }
✓ Correct! { get; private set; } means anyone can read, but only the class itself (its methods) can write. External code must go through Deposit()/Withdraw().
✗ Not quite. { get; private set; } gives public read access and private write access — exactly what we need for controlled modification through methods.

Reference Types — Two Variables, One Object

Objects are reference types. A variable holds a pointer, not the data itself:

BankAccount a = new BankAccount { Balance = 1000m };
BankAccount b = a;       // b points to the SAME object!

b.Balance = 2000m;
Console.WriteLine(a.Balance);  // 2000m — both see the change!
Variable a
→ ref
→
Object in Memory
2000m
←
Variable b
→ ref

No SQL equivalent. In SQL, every SELECT returns a copy of the data. In C#, two variables can point to the same object. If you need a copy, create a new object explicitly.

Quick Check

What does this code print?

BankAccount x = new BankAccount { Balance = 500m };
BankAccount y = x;
y.Balance = 0m;
Console.WriteLine(x.Balance);
A. 500
B. 0
C. Compiler error
D. NullReferenceException
✓ Correct! y = x copies the reference, not the object. Both x and y point to the same object, so changing it through y is visible through x.
✗ The answer is 0. Since y = x copies the reference (not the data), both variables point to the same object. Changing Balance through y changes it for x too.

Putting It All Together

public class BankAccount
{
    public long AccountId { get; }
    public string AccountNumber { get; }
    public string CCY { get; }
    public decimal Balance { get; private set; } = 0m;
    public decimal BlockedAmount { get; private set; } = 0m;
    public decimal AvailableBalance => Balance - BlockedAmount;  // computed

    public BankAccount(long id, string accNum, string ccy = "GEL")
    {
        if (string.IsNullOrEmpty(accNum)) throw new ArgumentException("Required.");
        AccountId = id; AccountNumber = accNum; CCY = ccy;
    }

    public void Deposit(decimal amount) { Validate(amount); Balance += amount; }
    public bool Withdraw(decimal amount)
    {
        Validate(amount);
        if (amount > AvailableBalance) return false;
        Balance -= amount; return true;
    }

    private void Validate(decimal amount)  // hidden helper
    {
        if (amount <= 0) throw new ArgumentException("Must be positive.");
    }
}

Live Demo — BankAccount in Action

var account = new BankAccount(1, "AC01GEL00000000012345");

account.Deposit(1000m);
Console.WriteLine(account.Balance);             // 1000

account.Deposit(500m);
Console.WriteLine(account.Balance);             // 1500

bool ok = account.Withdraw(200m);
Console.WriteLine($"{ok}, Balance: {account.Balance}");  // True, 1300

bool fail = account.Withdraw(5000m);
Console.WriteLine($"{fail}, Balance: {account.Balance}"); // False, 1300
Expected outputs
account.Balance → 1000.00 account.Balance → 1500.00 Withdraw(200) → True, Balance: 1300.00 Withdraw(5000) → False, Balance: 1300.00 account.AvailableBalance → 1300.00
