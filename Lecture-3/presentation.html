if / else — You Already Know This

T-SQL

IF @amount > 1000
BEGIN
    SET @fee = @amount * 0.01;
END
ELSE IF @amount > 100
BEGIN
    SET @fee = @amount * 0.02;
END
ELSE
BEGIN
    SET @fee = 5.00;
END

C#

if (amount > 1000)
{
    fee = amount * 0.01m;
}
else if (amount > 100)
{
    fee = amount * 0.02m;
}
else
{
    fee = 5.00m;
}
BEGIN...END becomes { } (curly braces). Condition must be in parentheses ( ).

Ternary & Null-Coalescing

IIF() → Ternary Operator ? :

-- T-SQL
SET @status = IIF(@balance >= 0,
                    'Active',
                    'Overdrawn');
// C#
string status = balance >= 0
    ? "Active"
    : "Overdrawn";

ISNULL/COALESCE → ??

-- T-SQL
SET @name = COALESCE(@first,
                      @last,
                      'Unknown');
SET @fee = @fee - ISNULL(@disc, 0);
// C#
string name = first ?? last ?? "Unknown";

decimal fee = fee - (discount ?? 0m);

// ??= assigns only if null
discount ??= 0m;

switch Expression — Beyond CASE WHEN

T-SQL CASE WHEN

SET @desc = CASE @type
    WHEN 'DEP' THEN 'Deposit'
    WHEN 'WIT' THEN 'Withdrawal'
    ELSE 'Unknown'
END;

C# switch expression Modern

string desc = type switch
{
    "DEP" => "Deposit",
    "WIT" => "Withdrawal",
    "TRF" => "Transfer",
    _     => "Unknown"     // _ = default case
};

It's an expression — it has a value, so you can assign it, return it, or use it inline. No break needed. The _ wildcard is the default.

Pattern Matching — SQL Has No Equivalent

C# switch expressions go beyond simple value matching:

// Match on value ranges
string risk = balance switch
{
    < 0                        => "Negative",
    >= 0 and < 1_000         => "Low",
    >= 1_000 and < 50_000   => "Medium",
    >= 50_000                => "High"
};

// Match on type (we'll use this more in Sessions 4-5)
string info = transaction switch
{
    DepositTransaction d    => $"Deposit of {d.Amount:C}",
    WithdrawalTransaction w => $"Withdrawal of {w.Amount:C}",
    null                     => "No transaction",
    _                        => "Unknown"
};
This is like a CASE expression that can also check ranges, types, and conditions simultaneously.

Quick Check

What is the C# equivalent of SQL's ISNULL(@discount, 0)?

A. discount.HasValue ? discount : 0m
B. discount ?? 0m
C. discount == null ? 0m : discount
D. NullCoalesce(discount, 0m)
✓ Correct! ?? is the null-coalescing operator — returns the left side if not null, otherwise the right side. Option C would also work, but ?? is idiomatic C#.
✗ Not quite. The null-coalescing operator ?? is the direct C# equivalent of ISNULL(x, default).

Loops — T-SQL Has One. C# Has Four.

LoopWhen to UseSQL Analogy
whileUnknown number of iterationsWHILE
forKnown count (0 to N)WHILE with counter variable
foreachIterate over a collectionCursor (but safe and automatic)
do-whileExecute at least onceWHILE 1=1 with BREAK inside

Golden Rule: Prefer foreach for collections. Use for when you need the index. Use while for condition-based loops.

while — Direct WHILE Translation

T-SQL

DECLARE @i INT = 1;
WHILE @i <= 10
BEGIN
    PRINT @i;
    SET @i = @i + 1;
END

C#

int i = 1;
while (i <= 10)
{
    Console.WriteLine(i);
    i++;   // same as i = i + 1
}

Increment Shortcuts

i++;    // i = i + 1
i--;    // i = i - 1
i += 5; // i = i + 5
i -= 3; // i = i - 3
i *= 2; // i = i * 2
i /= 4; // i = i / 4

for — When You Know the Count

// for (initializer; condition; increment)
for (int i = 0; i < 10; i++)
{
    Console.WriteLine(i);  // prints 0 through 9
}

① Initializer

int i = 0

Runs once before loop starts

② Condition

i < 10

Checked before each iteration

③ Increment

i++

Runs after each iteration

// Practical: generate 12 monthly statements
for (int month = 1; month <= 12; month++)
{
    Console.WriteLine($"Generating statement for month {month}");
}

foreach — Your Cursor Replacement

T-SQL Cursor (12 lines)

DECLARE @Id INT;
DECLARE cur CURSOR FOR
    SELECT AccountId FROM Accounts;
OPEN cur;
FETCH NEXT FROM cur INTO @Id;
WHILE @@FETCH_STATUS = 0
BEGIN
    -- process @Id
    FETCH NEXT FROM cur INTO @Id;
END;
CLOSE cur;
DEALLOCATE cur;

C# foreach (1 line)

foreach (int id in accountIds)
{
    // process id
}

foreach automatically opens, fetches, checks, and closes the collection iterator. No manual cleanup needed.

// With complex objects
foreach (var tx in transactions)
{
    Console.WriteLine(tx.Amount);
}

break and continue

break — Exit the Loop

foreach (var tx in transactions)
{
    if (tx.Amount > 100_000m)
    {
        Console.WriteLine(
            "Large transaction found.");
        break;  // exits the loop
    }
}

continue — Skip This Iteration

foreach (var tx in transactions)
{
    if (tx.Status == "Cancelled")
        continue; // skip, go to next

    ProcessTransaction(tx);
}
No direct T-SQL equivalent in WHILE loops — you'd use GOTO to a label at the end of the block, which is why cursor-based code is messy.

Quick Check

You need to iterate over a List<Transaction> and process each item. Which loop should you use?

A. while with a counter variable
B. for loop
C. foreach loop
D. do-while loop
✓ Correct! foreach is the natural choice for iterating over collections. It handles all the bookkeeping automatically — exactly like replacing a cursor with set-based SQL.
✗ foreach is the right answer. When you have a collection and want to process each element, foreach is cleaner, safer, and idiomatic C#.

Methods — Stored Procedures That Return Values

T-SQL Stored ProcedureC# Method
Named block of codeNamed block of code ✓
Has input parametersHas parameters ✓
OUTPUT parametersReturn type / out parameters
Cannot return a value directlyReturns any type directly ✓
Same name = errorOverloading: same name OK ✓
One result set maxReturns complex objects, tuples, lists ✓
A method is like a stored procedure that can directly hand you a value — no OUTPUT parameter, no SELECT result set, just: call it, get a value back.

Anatomy of a Method

// access modifier  return type   name          parameters
   public           decimal       CalculateFee  (decimal amount, string type)
{
    decimal fee = amount * 0.01m;
    return fee;    // hands the value back to the caller
}

Return Types

public void Log(int id)  { }   // nothing
public decimal Calc()   { }   // decimal
public bool IsValid()  { }   // true/false

Access Modifiers

public    // callable from anywhere
private   // only within this class
protected // class + subclasses
internal  // within this project

Parameters — Required and Optional

Required Parameters

-- T-SQL (must pass both)
CREATE PROCEDURE CalcFee
    @amount DECIMAL(18,2),
    @type   VARCHAR(10)
AS ...
// C# (must pass both)
public decimal CalcFee(
    decimal amount,
    string type)
{ ... }

Optional Parameters (Default Values)

-- T-SQL default value
CREATE PROCEDURE CalcFee
    @amount DECIMAL(18,2),
    @type   VARCHAR(10) = 'DEP'
AS ...
// C# default value
public decimal CalcFee(
    decimal amount,
    string type = "DEP")
{ ... }

CalcFee(100m);           // type = "DEP"
CalcFee(100m, "WIT");    // explicit

out — The OUTPUT Parameter

T-SQL OUTPUT

CREATE PROCEDURE ProcessPayment
    @amount  DECIMAL,
    @txId    INT OUTPUT,
    @fee     DECIMAL OUTPUT
AS
BEGIN
    SET @txId = NEXT VALUE FOR seq;
    SET @fee  = @amount * 0.01;
END;

C# out

public bool ProcessPayment(
    decimal amount,
    out int txId,
    out decimal fee)
{
    txId = GenerateNextId();
    fee  = amount * 0.01m;
    return true;
}

// Calling it:
ProcessPayment(500m, out int id, out decimal f);
// Discard if not needed:
ProcessPayment(500m, out int id, out _);

Compiler-enforced: out parameters must be assigned inside the method before it returns. If you forget, it won't compile. T-SQL has no such guarantee.

The TryParse Pattern — out in Action

The most common use of out in everyday C# code:

T-SQL

DECLARE @result INT;
SET @result = TRY_CAST(@input AS INT);
IF @result IS NULL
    PRINT 'Not valid';

C#

if (int.TryParse(userInput, out int result))
{
    Console.WriteLine($"Parsed: {result}");
}
else
{
    Console.WriteLine("Not a valid integer");
}

TryParse is the pattern where the method returns bool (success/failure) and gives you the parsed value via out. You'll see this everywhere: decimal.TryParse, DateTime.TryParse, Guid.TryParse.

Method Overloading — Same Name, Different Signatures

In T-SQL, two procedures with the same name would error. In C#, same name + different parameters = overloading.
// Three overloads — compiler picks the right one
public decimal CalculateFee(decimal amount)
    => CalculateFee(amount, "STANDARD");    // delegate to full version

public decimal CalculateFee(decimal amount, string type)
{
    return type switch
    {
        "EXPRESS" => amount * 0.02m,
        "VIP"     => 0m,
        _          => amount * 0.01m
    };
}

public decimal CalculateFee(decimal amount, string type, bool isWeekend)
{
    decimal fee = CalculateFee(amount, type);
    return isWeekend ? fee * 1.5m : fee;
}
CalculateFee(100m);                       // → calls overload 1
 CalculateFee(100m, "EXPRESS");            // → calls overload 2
 CalculateFee(100m, "EXPRESS", true);     // → calls overload 3

Modern Method Syntax

Expression-Bodied Methods

When the method body is a single expression, use => shorthand:

// Full method
public decimal CalculateVat(decimal amount)
{
    return amount * 0.20m;
}

// Expression-bodied (identical result)
public decimal CalculateVat(decimal amount) => amount * 0.20m;

public bool IsPositive(decimal amount) => amount > 0;
public string FormatAmount(decimal amount) => $"{amount:C}";

Return a Tuple Instead of out

// Modern preference: return a tuple instead of out parameters
public (decimal Fee, string Description) Calculate(decimal amount, string type)
{
    decimal fee = amount * 0.01m;
    string desc = $"Fee for {type}: {fee:C}";
    return (fee, desc);
}

var (fee, desc) = Calculate(500m, "WIT");   // destructure the tuple

Quick Check

A C# out parameter must be assigned inside the method before it returns. What happens if you forget?

A. It defaults to null or zero at runtime
B. The code won't compile — it's a compiler error
C. It throws a NullReferenceException at runtime
D. T-SQL's behavior: it stays as the value it had before the call
✓ Correct! The C# compiler enforces that all code paths must assign out parameters. This catches bugs at compile time that T-SQL would only reveal at runtime (or not at all).
✗ Not quite. The C# compiler requires all out parameters to be assigned before the method returns — it's a compile-time error, not a runtime error.

The Full Transformation

T-SQL Procedure

CREATE PROCEDURE usp_CalcFee
    @amount  DECIMAL,
    @type    VARCHAR(10) = 'STD',
    @fee     DECIMAL OUTPUT
AS BEGIN
    IF @type = 'EXP'
        SET @fee = @amount * 0.02;
    ELSE IF @type = 'VIP'
        SET @fee = 0;
    ELSE
        SET @fee = @amount * 0.01;
END;

C# Method

public decimal CalcFee(
    decimal amount,
    string type = "STD")   // default value
{
    return type switch
    {
        "EXP" => amount * 0.02m,
        "VIP" => 0m,
        _      => amount * 0.01m
    };
    // no OUTPUT, returns directly
}

The C# version: returns value directly, uses switch expression, has a default parameter, is shorter and clearer.

static decimal CalculateBaseFee(decimal amount, string transType)

        "WIT" => amount switch
