The Problem — Repeated Structure

In HSP, many tables share the same columns:

CREATE TABLE hw.Accounts (
    AccountID BIGINT, CustomerID INT, CCY VARCHAR(5),
    Balance DECIMAL, AccountState BIGINT, CreateTime DATETIME2
);
CREATE TABLE hw.Deposits (
    DepositID BIGINT, CustomerID INT, CCY VARCHAR(5),
    Balance DECIMAL, AccountState BIGINT, CreateTime DATETIME2,
    InterestRate DECIMAL, MaturityDate DATETIME2   -- extra
);
CREATE TABLE hw.Loans (
    LoanID BIGINT, CustomerID INT, CCY VARCHAR(5),
    Balance DECIMAL, AccountState BIGINT, CreateTime DATETIME2,
    InterestRate DECIMAL, RemainingDebt DECIMAL     -- extra
);

6 columns are identical across all three tables. In SQL, you can't avoid this repetition. In C#, you can.

Inheritance — Extending a Base Class

The Base Class

public class FinancialProduct
{
    public long Id { get; }
    public int CustomerId { get; }
    public string CCY { get; }
    public decimal Balance { get; protected set; }
    public DateTime CreateTime { get; }

    public FinancialProduct(
        long id, int custId, string ccy)
    {
        Id = id; CustomerId = custId;
        CCY = ccy; CreateTime = DateTime.Now;
    }
}

Derived Classes

public class DepositAccount
    : FinancialProduct
{
    public decimal InterestRate { get; }
    public DateTime MaturityDate { get; }

    public DepositAccount(
        long id, int custId, string ccy,
        decimal rate, DateTime maturity)
        : base(id, custId, ccy)
    {
        InterestRate = rate;
        MaturityDate = maturity;
    }
}
Inheritance is like a view that extends a base table. The view inherits all columns and can add its own computed columns.

: base(...) — Calling the Parent Constructor

When creating a derived object, the parent constructor runs first:

public class LoanAccount : FinancialProduct
{
    public decimal InterestRate { get; }
    public decimal RemainingDebt { get; private set; }

    public LoanAccount(long id, int custId, string ccy,
                       decimal rate, decimal debt)
        : base(id, custId, ccy)      // Step 1: set up FinancialProduct
    {
        InterestRate = rate;          // Step 2: set up LoanAccount extras
        RemainingDebt = debt;
    }
}

base(id, custId, ccy) calls the parent's constructor. It's like inserting into the base table first, then into the extension table.

virtual and override

A base class marks methods as virtual = "derived classes can change this behavior":

Base — Default Behavior

public class FinancialProduct
{
    // virtual = CAN be overridden
    public virtual string GetStatement()
    {
        return $"Balance: {Balance:N2} {CCY}";
    }
}

Derived — Custom Behavior

public class DepositAccount
    : FinancialProduct
{
    // override = I'm REPLACING it
    public override string GetStatement()
    {
        return $"Deposit: {Balance:N2}"
             + $" @ {InterestRate:P2}";
    }
}
FinancialProduct p = new DepositAccount(1, 42, "GEL", 0.12m, maturity);
Console.WriteLine(p.GetStatement());  // "Deposit: 0.00 @ 12.00%" — DepositAccount version runs!

abstract — "You MUST Implement This"

An abstract method has no body — derived classes must provide one:

public abstract class Transaction
{
    public long Id { get; }
    public decimal Amount { get; }

    protected Transaction(long id, decimal amount) { Id = id; Amount = amount; }

    public abstract string TransactionType { get; }  // no body — MUST override
    public abstract bool Execute();                   // no body — MUST override
}

// var t = new Transaction(1, 100m);  // COMPILER ERROR — abstract!

public class DepositTransaction : Transaction
{
    public override string TransactionType => "DEPOSIT";  // required
    public override bool Execute() { /* deposit logic */ return true; }
    public DepositTransaction(long id, decimal amount) : base(id, amount) { }
}
An abstract class is like a table type definition that can never hold data itself — only defines the structure for others to follow.

Quick Check

Which keyword means "derived classes must provide their own implementation"?

A. virtual
B. abstract
C. override
D. sealed
✓ Correct! abstract means "no default implementation — every derived class must provide one." virtual means "here's a default, but you CAN override it."
✗ Not quite. abstract = must implement. virtual = can override (optional). override = the act of replacing. sealed = prevent further inheritance.

Interfaces — Contracts for Behavior

An interface defines what a class must do, not how:

public interface IFeeCalculator
{
    decimal Calculate(decimal transactionAmount);
    string FeeType { get; }
}

Any class that "implements" this interface promises to have Calculate and FeeType:

public class FixedFeeCalculator : IFeeCalculator
{
    public string FeeType => "Fixed";
    public decimal Calculate(decimal amount) => 5m;  // always 5
}

public class PercentFeeCalculator : IFeeCalculator
{
    public string FeeType => "Percent";
    public decimal Calculate(decimal amount) => amount * 0.02m;  // 2%
}
An interface is like a rule: "every fee procedure must accept @Amount DECIMAL and return DECIMAL." Multiple procedures can follow the same rule differently.

The HSP Problem — IF/ELSE Chains

SQL — Growing IF/ELSE

-- Every new fee type = another ELSE IF
IF @FeeType = 1
    RETURN @FixedFee
ELSE IF @FeeType = 2
    RETURN @Amount * @Percent / 100
ELSE IF @FeeType = 3
    -- tiered logic...
ELSE IF @FeeType = 4
    -- new type...
ELSE IF @FeeType = 5
    -- another new type...

C# — Interface

// No IF/ELSE — just use the interface
IFeeCalculator calc = GetCalculator(feeConfigId);
decimal fee = calc.Calculate(amount);

// Adding a new fee type?
// Just create a new class.
// No existing code changes!
public class VolumeFeeCalculator
    : IFeeCalculator
{
    public decimal Calculate(decimal amount)
        => /* new logic */;
}

Open/Closed Principle: Open for extension (new classes), closed for modification (existing code doesn't change).

Multiple Interfaces — Multiple Contracts

A class can implement many interfaces (unlike inheritance — only one base class):

public interface ILoggable     { string ToLogString(); }
public interface IAuditable    { DateTime LastModified { get; } }
public interface IReversible   { bool Reverse(); }

// One class, three contracts
public class WireTransfer : Transaction, ILoggable, IAuditable, IReversible
{
    public string ToLogString() => $"Wire #{Id}: {Amount:N2}";
    public DateTime LastModified { get; }
    public bool Reverse() { /* reversal logic */ return true; }
    // ...plus Transaction members
}

Inheritance: one base class (IS A). Interfaces: many contracts (CAN DO).

Polymorphism — One Call, Many Behaviors

Polymorphism = treating different objects through a common type. Each responds with its own behavior:

HSP Way (IF/ELSE)

IF @Type = 'DEPOSIT'
    EXEC hp.spDeposit @ID
ELSE IF @Type = 'WITHDRAWAL'
    EXEC hp.spWithdraw @ID
ELSE IF @Type = 'TRANSFER'
    EXEC hp.spTransfer @ID
ELSE IF @Type = 'FEE'
    EXEC hp.spChargeFee @ID

C# Way (Polymorphism)

// No IF/ELSE needed!
Transaction tx = GetTransaction(id);
tx.Execute();
// The correct version runs automatically
// based on the actual type of the object

// Process a whole batch:
foreach (var tx in batch)
    tx.Execute();
// Each executes its own way

That's polymorphism: one Execute() call, but deposits deposit, withdrawals withdraw, and transfers transfer. The object knows what to do.

Polymorphism Demo

List<Transaction> batch = new()
{
    new DepositTransaction(1, 1000m, "GEL", account),
    new WithdrawalTransaction(2, 200m, "GEL", account),
    new TransferTransaction(3, 300m, "GEL", account, otherAccount)
};

foreach (Transaction tx in batch)
{
    bool ok = tx.Execute();
    Console.WriteLine($"{tx.TransactionType} #{tx.Id}: {(ok ? "OK" : "FAIL")}");
}
Expected output
DEPOSIT #1: OK WITHDRAWAL #2: OK TRANSFER #3: OK account.Balance = 500.00 GEL

Polymorphism Through Interfaces

List<IFeeCalculator> calculators = new()
{
    new FixedFeeCalculator(5m),
    new PercentFeeCalculator(2m, 1m, 100m),
    new TieredFeeCalculator()
};

decimal amount = 5000m;
foreach (IFeeCalculator calc in calculators)
{
    Console.WriteLine($"{calc.FeeType}: {calc.Calculate(amount):N2}");
}
Expected output
Fixed: 5.00 Percent: 100.00 Tiered: 30.00
Same Calculate() call, three different results. The interface is the contract; polymorphism is the magic.

Quick Check

What is the main advantage of polymorphism over IF/ELSE chains?

A. It runs faster
B. It uses less memory
C. Adding new types doesn't require changing existing code
D. It looks more professional
✓ Correct! With IF/ELSE, every new type means modifying existing code. With polymorphism, you just add a new class — existing code keeps working untouched.
✗ The key benefit is extensibility without modification. New types = new classes, not new IF branches. This is the Open/Closed Principle in action.

Strategy Pattern — Swappable Behavior

The combination of interfaces + polymorphism creates the Strategy Pattern:

public class TransactionProcessor
{
    private readonly ITransactionValidator _validator;
    private readonly IFeeCalculator _feeCalc;

    public TransactionProcessor(ITransactionValidator v, IFeeCalculator f)
    { _validator = v; _feeCalc = f; }

    public bool Process(Transaction tx, BankAccount acct)
    {
        if (!_validator.Validate(tx, acct)) return false;
        decimal fee = _feeCalc.Calculate(tx.Amount);
        return tx.Execute();
    }
}

// Retail: percent fee, basic validation
var retail = new TransactionProcessor(new BasicValidator(), new PercentFeeCalculator(...));

// VIP: no fee, strict validation
var vip = new TransactionProcessor(new StrictValidator(), new FixedFeeCalculator(0m));

Same class, completely different behavior — just by injecting different implementations. This is how ASP.NET Core services work.

is and as — Type Checking

Sometimes you need to know the actual type of a polymorphic object:

Transaction tx = GetTransaction(id);

// Pattern matching with 'is'
if (tx is TransferTransaction transfer)
{
    Console.WriteLine($"From: {transfer.SourceAccount.AccountNumber}");
    Console.WriteLine($"To:   {transfer.DestinationAccount.AccountNumber}");
}
else if (tx is DepositTransaction deposit)
{
    Console.WriteLine($"Into: {deposit.TargetAccount.AccountNumber}");
}
Like checking a TransactionType column: IF @Type = 'TRANSFER' SELECT SourceAccount, DestAccount ... — but type-safe.

sealed — Locking Down

sealed prevents a class from being inherited further:

public sealed class FeeChargeTransaction : Transaction
{
    public override string TransactionType => "FEE";
    public override bool Execute() { /* fee logic */ return true; }
}

// This would NOT compile:
// public class SpecialFee : FeeChargeTransaction { }  // ERROR!

Quick Check

A class needs to follow multiple contracts: it must be loggable, auditable, AND reversible. What should you use?

A. Inherit from three base classes
B. Implement three interfaces
C. Create one base class with all three behaviors
D. Use static methods for each capability
✓ Correct! C# supports single inheritance (one base class) but multiple interface implementation. Use interfaces for capabilities: class WireTransfer : Transaction, ILoggable, IAuditable, IReversible.
✗ C# only allows one base class (single inheritance), but a class can implement many interfaces. Use interfaces for "can do" capabilities.
